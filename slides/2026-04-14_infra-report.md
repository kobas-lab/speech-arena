# SpeechArena インフラ運用報告

LLM-jp Moshi 音声対話モデルの A/B 評価基盤

2026-04-14

---

## 今日お伝えしたいこと

1. **現状の構成**: 研究室GPU ベース、動作確認完了
2. **課題**: 研究室GPU ではワーカーが**1人ずつ**しか評価できない
3. **選択肢**: AWS への移行を検討したいが、**技術面とコスト面でトレードオフ**がある
4. **ご相談**: クラウドソーシング規模に合わせてどう段階的に進めるか

---

## 現状の構成

```
ワーカー (ブラウザ)
  ↓ HTTPS
Vercel (Next.js, 無料)
  ├─ フロント: https://web-fawn-five-91.vercel.app
  └─ DB: Supabase PostgreSQL (無料)
  ↓ Cloudflare Tunnel (HTTPS)
研究室サーバー (RTX 3090 × 2)
  ├─ moshi-server A (ポート 8998) ← Model A
  └─ moshi-server B (ポート 8999) ← Model B
```

- 6種類の LLM-jp Moshi モデルのうち 2つを常時起動
- 評価セッションで **A×2会話 → B×2会話 → 投票** の流れ（所要 約10分）
- FD-DMOS 4指標 + 成功/失敗 + パケットロス で定量評価

---

## 課題: 同時 1 ワーカーしか捌けない

- RTX 3090 は 24GB VRAM → **1枚で1モデルが上限**
- 2枚 = 2モデル同時起動が物理上の上限
- **ワーカーが会話中は GPU 占有**、2人目は待機

### クラウドソーシングの現実的影響
| ワーカー数 | 所要時間（1セッション 10分） |
|---|---|
| 10人 | 100分（1.7時間） |
| 50人 | 8.3時間 |
| 100人 | 16.7時間 |

<span class="danger">→ 規模を上げるには並列化が必須</span>

---

## 選択肢A: AWS に全面移行

### 技術面
- **オートスケーリング**: ワーカー数に応じて GPU インスタンスを自動追加
- **Warm Pool**（常時起動プール）: 2人目以降の待ち時間を最小化（3-5分 → 即時）
- 6モデルから**ランダム2モデル**を動的に割当可能

### ボトルネック
- Moshi モデルのロードに **コールドスタート 15-20分**
- ウォームスタートでも 3-5分
- → **Warm Pool が UX 改善の鍵**

---

## AWS GPU インスタンスの選定結果

2モデル同時起動に必要な VRAM: **34GB**

| インスタンス | VRAM | 同時モデル数 | 備考 |
|---|---|---|---|
| g5.xlarge (A10G 24GB) | 24GB | 1 | 1モデルのみ |
| g5.2xlarge (A10G 24GB) | 24GB | 1 | 1モデルのみ |
| g6e.xlarge (L40S 48GB) | **46GB** | **2** | ✅ 採用 |

- 東京リージョン: **Spot GPU の vCPU quota = 0**（利用不可）
- → <span class="highlight">us-east-1 で On-demand 運用</span>

---

## コスト: AWS で GPU を常時起動した場合

g6e.xlarge (us-east-1, on-demand) = **$1.86/h**

| 運用パターン | GPU 時間/月 | 月額 |
|---|---|---|
| 評価時のみ（1日2h × 週5） | 44h | **~$96** |
| 勤務時間帯のみ（8h × 週5） | 176h | **~$336** |
| 常時起動（Warm Pool 1台） | 720h | <span class="danger">**~$1,340**</span> |
| 常時起動 × 2台（並列対応） | 1,440h | <span class="danger">**~$2,680**</span> |

+ EBS 200GB = $16/月、データ転送・S3 数ドル

---

## 技術課題とコストのトレードオフ

| 要件 | 技術手段 | 月額コスト目安 |
|---|---|---|
| 1人ずつ捌ければ良い | 研究室GPU | **$0**（電気代のみ） |
| 学外公開、数人並列 | AWS オンデマンド（使う時だけ起動） | ~$100-300 |
| <span class="highlight">UX 重視・待ち時間ゼロ</span> | Warm Pool 1台常時起動 | ~$1,340 |
| 大規模クラウドソーシング | ASG + Warm Pool 複数台 | ~$2,000+ |

→ <span class="danger">「何人同時を許容するか」「どの程度の待ち時間を許容するか」で選択</span>

## 提案: 段階的移行ロードマップ

### Phase 1 — 今すぐ（学内クラウドソーシング）
- **研究室GPU のまま** 1ペアずつ評価データ収集
- Bradley-Terry 実装で全ペア網羅不要に
- コスト: **$0**

### Phase 2 — 学外公開準備（2-3週）
- AWS に移行、認証追加、動的モデル割当復活
- **使う時だけ起動**（自動停止スケジューラ）
- コスト: **~$100-300/月**

### Phase 3 — 本格運用（必要に応じて）
- Warm Pool + Auto Scaling で待ち時間ゼロ
- コスト: **~$1,340-2,680/月**

---

## 運用案: 募集期間に合わせた弾力運用

### 基本方針
- **通常期間**（開発・検証）: 日中 8時間のみ起動 or コールドスタート（使う時だけ）
- **クラウドソーシング募集期間**: 常時起動 GPU を**複数台に増やし、並列4-5人**捌けるようにする

### ポイント
- Warm Pool 1台 = 同時1-2人 → **募集期間だけ ASG でスケールアップ**
- 理想: 並列 4-5 人対応（g6e.xlarge を 3-4 台常時起動）
- 募集終了後は 1台 or 自動停止に戻す

---

## 運用案: コスト見積もり

| 運用モード | GPU 台数 | 想定月額 | 捌ける並列数 |
|---|---|---|---|
| 通常期（コールドスタート） | 0（必要時起動） | ~$50 | 1（待ち3-5分） |
| 通常期（日中8h × 週5） | 1 | ~$336 | 1-2 |
| <span class="highlight">募集期間（Warm Pool 複数台）</span> | 3-4 | **~$4,000-5,400** | **4-5人並列** |

**年間イメージ**（通常期 10ヶ月 + 募集期 2ヶ月の場合）
- 通常期: $336 × 10 = ~$3,360
- 募集期: $5,000 × 2 = ~$10,000
- **年間合計: 約 $13,000-14,000**

→ <span class="danger">募集期間をどの程度の頻度・期間で設定するかがコストの肝</span>
